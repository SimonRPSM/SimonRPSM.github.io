## I Don't Know What Anyone is Talking About
The world of Software is filled with pretty arcane terminology that can be pretty daunting the non-initiated. I would like to taxonomize this lexicon in the following pieces, also for the non-initiated:

1. Computer Science Terminology
2. Internal company jargon
3. Specific frameworks, tools, etc
4. Software Engineering methodologies, concepts, principles
5. Software Engineering Methodologies (bear with me)


The first is reasonably self-explanatory. You can learn the lexicon of computer science by ... learning computer science. For example, you can open up the Wizard Book (Structure and Interpretation of Computer Programs) and learn about recursion. Learning this kind of stuff will mean that when someone mentions implementing a Depth-first search algorithm for something you'll nod and say that it sounds like a good idea because it actually does, not because you do not want to say that you know "some of these words".

The second is also self-explanatory. Every company creates a vast amount of internal terminology. Servers have names, libraries have names, internal methodologies and processes have names, etc. Learning these requires going over the internal documentation (or tribal knowledge) of the company until things stick.

The third one is never-ending. There is always a new framework, a new tool. The thing that you are currently using is always being heralded as dead by someone in a blog like this one because the thing that you are not yet using that just came out of alpha is amazing. But it is worth trying to keep up with the things out there via blogs, publications, etc. That way you may be able to find stuff that can save you time. An example is the emergence of more and more Continuous Integration (CI) tools that can save you from needing to manually run your own test.  

Things get a bit more interesting in attempting to separate the last few items.

There are some Methodologies with a capital M that attempt to describe the holistic process (or pieces of this process) of creating software. For example, if someone mentions the *Dynamic Systems Development Method*, it refers to a specific way to develop software that's an attempt to mix waterfall and agile methodologies (I'm oversimplifying that).

This is slightly different than concepts or methodologies like the entirety of the book *Clean Code*, for example, that describes a whole plethora of things that are not quite a "system" but a series of recommendations on how to build software. Similarly, SOLID is a conglomeration of design principles that tell you how to think about writing the software, but they are not really a Methodology.

\- Simon Perez Santa Maria
