# I have been thinking about the process it takes to spin out a new team

And I have a lot of trouble thinking about the different pieces that need to be created. In order to organize my own thoughts, I have been separating them into three layers of abstraction.

# Project processes

At the highest level lie the tools, processes and methodologies that direct how work should be scheduled. It is at this level that we have DSDM, Agile, Waterfall, etc.

This generally lies in the realm of the Project Manager. It is their subject matter expertise that is used to figure out the way that work should be planned and scheduled. This is done with the input of the technical lead - but input means input.

We can think of this as the month-to-month things that the team needs to run.

# Development Processes

At this level we start discussing the specific workflows used to meet the demands of the business.

For example, here we investigate the type of version control needed (it's needed, that's not part of the discussion), the artifacting strategy, the documentation strategy.

It is also here that we consider the processes for working as a team - does GitFlow Lite make sense? Should we all just work on separate repos and communicate our artifacts via carrier pigeons?

# Development Craftsmanship

I really like the Clean Code terminology for this. At this level we discuss what the system looks like and how to actually get it done.

At this level we discuss whether we want to create the thing in a Functional Language, rely on SOLID OOP principles, or code everything in assembly using only nano as the editor.
